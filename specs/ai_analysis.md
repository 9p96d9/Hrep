## 3. `detail_street` モード仕様（正式版）

### 設計意図

旧`detail`モードの問題: `hand_reading` が全ストリートを1〜2文に圧縮するため、
AIがどのストリートの話をしているか不明確になり、誤ったMDF言及が起きやすい。

解決策: ストリートを独立したフィールドに分割し、各フィールドのMDFルールを構造で強制する。

### 出力フィールド定義

JSON配列で返す。各ハンドに以下のフィールドが必須。

```json
{
  "id": 1,
  "preflop_context": "Hero位置・RFI/3bet/Defend・レンジ概要（1〜2文）",
  "flop_read": "Flop board + Heroレンジ保持度 + 相手レンジ変化（1〜2文）",
  "turn_read": "Turn card + レンジ変化（1〜2文）。到達していない場合は空文字\"\"",
  "river_analysis": "River decision + [GTO数学]のMDF引用（1〜3文）。到達していない場合は空文字\"\"",
  "opp_exploit": "相手のGTOからの逸脱と搾取戦略（アクション名で）",
  "kaizen": "代替ライン or 「このラインで十分」"
}
```

### フィールド別 MDF ルール

| フィールド | MDF 言及 | 理由 |
|---|---|---|
| `preflop_context` | 禁止 | エクイティ未確定 |
| `flop_read` | 禁止 | エクイティが残る |
| `turn_read` | 禁止 | エクイティが残る |
| `river_analysis` | 許可（[GTO数学]の値のみ引用） | エクイティ確定ストリート |
| `opp_exploit` | 禁止 | MDFは搾取の軸ではない |
| `kaizen` | 禁止 | — |

### バッチプロンプト指示（システムプロンプトに含める）

```
フィールドはストリートごとに分かれています。
各フィールドは該当ストリートの話のみ書いてください。
turn_read / river_analysis は、ハンドがそのストリートに到達していない場合は空文字を返してください。
MDF は river_analysis フィールドのみで言及してください。他のフィールドでは一切触れないこと。
```

## 4. `detail` モード（旧形式・移行期間のみ）

JSON配列で返す。各ハンドに以下のフィールドが必須。

| フィールド | 型 | 定義 |
|---|---|---|
| `id` | int | ハンド番号（入力のidと一致） |
| `spot_range` | string | このポジション/分類/ボードでHeroが均衡上持ちうるレンジ概要（1〜2文） |
| `balance_note` | string | [GTO数学]の数値をポジション・ストリート・相手テンデンシーで補正した均衡コメント（1〜2文） |
| `hand_reading` | string | 各ストリートで相手レンジがどう変化したか（1〜2文） |
| `opp_exploit` | string | 相手のGTOからの逸脱と具体的な搾取戦略（アクション名で） |
| `rep` | string | Heroが表現できるハンド2〜3個（例: ハートフラッシュ・セット88） |
| `kaizen` | string | 代替ライン or 「このラインで十分」 |

### `detail` → `detail_street` の差分サマリー

| 項目 | `detail` | `detail_street` |
|---|---|---|
| `hand_reading` | ✅（全ストリート圧縮） | 削除（ストリート別フィールドに分割） |
| `balance_note` | ✅ | 削除（各ストリートに分散） |
| `spot_range` | ✅ | 削除（`preflop_context` に統合） |
| `rep` | ✅ | 削除（street 文脈で自然に語られる） |
| `preflop_context`〜`river_analysis` | なし | ✅ |
| `opp_exploit` / `kaizen` | ✅ | ✅（同じ） |

## 5. explain モード — 出力形式

フリーテキスト（JSON不要）。400〜1200文字。

必須セクション（段落で展開）:
1. 均衡レンジとHeroのポジション（OOP/IP）
2. GTO数学的観点（MDF/必要成功率/バリューターゲット）
3. 相手レンジの変化と読み
4. 相手への搾取戦略
5. 代替ライン

## 6. 品質ルール（全モード共通・厳守）

### 主語ルール
- 主語は必ずポジション名にする（CO, BB, BTN 等）
- ❌ 「ヒーローはコールした」「相手はベットした」
- ✅ 「BBはBTNのCBetに対してコールした」

### 数値ルール
- `[GTO数学]` ブロックに記載の数値のみ引用可
- AIが自前で計算・推定した数値は一切使用禁止
- 数値が不明な場合は言及しない

### MDF適用ルール
- **リバーのみ** MDF = 1 - α に言及する
- フロップ/ターンではMDFに触れない
- フロップ/ターンはエクイティ・ドロー・レンジ構成で論じる

### レンジ表記ルール
- 具体的なレンジ記法を使う（例: ATo+, KQs, 22-44, TT-JJ）
- 「強いハンド」「弱いドロー」などの曖昧表現を避ける

### 評価軸ルール
- 個別ハンドの勝敗ではなく「このスポットでHeroのレンジは均衡しているか」を軸にする
- 「良い/悪い/ミス」という結果論的評価を出さない
- OOPはMDF目安より多めのフォールドが均衡に近い場合がある（定義は変わらない）

## 7. [GTO数学]ブロック仕様

プロンプトに含まれる事前計算済みコンテキスト。`_compute_gto_math()`が生成（`specs/gto_math.md` §2参照）。

```
[GTO数学] {スポットタイプ} | {ストリート} | {数値情報}
```

AIはこのブロックの数値を再計算せず、そのまま引用して解釈する。

## 8. 受入基準（= テストケース）

### `detail_street` モード（主）

| 確認項目 | 合格条件 |
|---|---|
| 主語がポジション名 | 全フィールドで "CO", "BB", "BTN" 等で始まる文を含む |
| flop_read に MDF なし | "MDF" 文字列が `flop_read` に含まれない |
| turn_read に MDF なし | "MDF" 文字列が `turn_read` に含まれない |
| river 未到達 → 空文字 | フロップ終了ハンドで `turn_read = ""`, `river_analysis = ""` |
| river 到達 → MDF あり | `river_analysis` が非空かつ "MDF" または "必要エクイティ" を含む |
| 数値の出典 | AI独自計算数値が0件（[GTO数学]のみ） |
| opp_exploit が具体的 | アクション名（"bet", "raise", "fold" 等）を含む |
| 全フィールドの存在 | 6フィールド全て存在（未到達ストリートは空文字で存在） |

### `detail` モード（移行期間のみ）

| 確認項目 | 合格条件 |
|---|---|
| 主語がポジション名 | "CO", "BB", "BTN" 等で始まる文を含む |
| MDF言及ストリート | riverのスポットのみ "MDF" 文字列が含まれる |
| 数値の出典 | [GTO数学]ブロックの値と一致、またはAI独自数値が0件 |
| 全フィールドの存在 | spot_range/balance_note/hand_reading/opp_exploit/rep/kaizen 全て非空 |
| レンジ表記 | "+" や "-" を含む具体的なレンジ記法が spot_range に含まれる |

> このテストはAI APIを呼ばない。解析済みJSONを `tests/fixtures/` に置いて検証する（`tests/PLAN.md`参照）。
